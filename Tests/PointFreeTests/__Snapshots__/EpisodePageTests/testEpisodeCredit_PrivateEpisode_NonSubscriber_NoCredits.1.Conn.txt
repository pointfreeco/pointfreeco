              <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">We’ve now had an entire episode just about functions where we emphasized the importance of looking at the input and output types of functions to understand how they compose, but there are a lot of things a function can do that aren’t captured by its signature alone. These things are called “side effects.”
              </p>
              <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Side effects are one of the biggest sources of complexity in code, and to make things worse they’re very difficult to test and they don’t compose well. We’ve seen from the last episode that we get a lot of benefits from embracing function composition, but side effects throw a wrench in that.
              </p>
              <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">In this episode we’ll cover a few kinds of side effects, show why they’re so difficult to test, why they don’t compose, and try to address these problems in a nice way.
              </p>
              <div>
                <div>
                  <div class="font-variant-numeric-vLOwL2 line-height-B4ojl2 margin-left-xuQdp2 position-9Qeuv2 text-align-mMP5l4 width-TNVhF4 font-weight-evfWi1 line-height-9Qkso4 font-size-RTPM71" id="t43"><a data-timestamp="43" class="text-decoration-Mif7n4 text-decoration-Q5Er64 text-decoration-Zf0gY color-Vv8Gw4 color-9rBiP color-x5AlI2 color-CwERD1 color-KmPrN color-lJpf9" href="#t43">0:43</a>
                  </div>
                </div>
              </div>
              <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Side effect is a pretty overloaded term, so in order to define it, let’s first look at a function that has no side effects:
              </p>
              <pre class="margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">func compute(_ x: Int) -> Int {
  return x * x + 1
}
</code>
              </pre>
              <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">When we call our function, we get a result back:
              </p>
              <pre class="margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">compute(2)  // 5
</code>
              </pre>
              <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">A really nice property of functions without side effects is that no matter how many times we call one with the same input, we always get the same output:
              </p>
              <pre class="margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">compute(2)  // 5
compute(2)  // 5
compute(2)  // 5
</code>
              </pre>
              <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">This predictability makes it incredibly simple to write tests for them.
              </p>
              <pre class="margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">assertEqual(5, compute(2))  // ✅
</code>
              </pre>
              <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">If we ever write a test with the wrong expectation or the wrong input for an expectation, it will <em>always</em> fail.
              </p>
              <pre class="margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1" data-line="1-2"><code class="language-swift highlight-fail">assertEqual(4, compute(2))
assertEqual(5, compute(3))
</code>
              </pre>
              <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Let’s add a side effect to our function.
              </p>
              <pre class="margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">func computeWithEffect(_ x: Int) -> Int {
  let computation = x * x + 1
  print("Computed \(computation)")
  return computation
}
</code>
              </pre>
              <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">We’ve inserted a <code>print</code> statement right in the middle.
              </p>
              <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">When we call <code>computeWithEffect</code> with the same input as before, we get the same output:
              </p>
              <pre class="margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">computeWithEffect(2)  // 5
</code>
              </pre>
              <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">But if we look at our console, there’s some <em>additional</em> output here.
              </p>
              <pre class="margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-txt">Computed 5
</code>
              </pre>
              <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">If we compare function signatures, <code>computeWithEffect</code> is exactly the same as <code>compute</code>, but work is being done that we couldn’t have accounted for by looking at the signature alone. The <code>print</code> function is reaching out into the world and making a change, in this case, printing to our console. Side effects require understanding the body of the function to know they’re hiding in there.
              </p>
              <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Let’s write a test for this function:
              </p>
              <pre class="margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1" data-line="1"><code class="language-swift highlight-pass">assertEqual(5, computeWithEffect(2))
</code>
              </pre>
              <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">It passes! But now we have an <em>additional</em> line in our console.
              </p>
              <pre class="margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-txt">Computed 5
Computed 5
</code>
              </pre>
              <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">And this is behavior that we just can’t test. Here we’re just printing to the console, so it may not seem like a big deal, but if we swapped printing out for another effect, like writing to disk, making an API request, or analytics tracking, we start to care more that this behavior is happening and that we can test it.
              </p>
              <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Side-effects can also break compositional intuitions we build. In our episode on functions, we discussed how mapping over an array with two functions is the same as mapping over an array with the composition of those functions:
              </p>
              <pre class="margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">[2, 10].map(compute).map(compute)  // [26, 10202]
[2, 10].map(compute >>> compute)   // [26, 10202]
</code>
              </pre>
              <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Now let’s try that with <code>computeWithEffect</code>:
              </p>
              <pre class="margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">[2, 10].map(computeWithEffect).map(computeWithEffect)
// [26, 10202]
[2, 10].map(computeWithEffect >>> computeWithEffect)
// [26, 10202]
</code>
              </pre>
              <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">The return values are equal, but when we look at the console, the behavior is not!
              </p>
              <pre class="margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-txt">Computed 5
Computed 101
Computed 26
Computed 10202
--
Computed 5
Computed 26
Computed 101
Computed 10202
</code>
              </pre>
              <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">We can no longer take advantage of this property without having to consider the side effects. Our ability to make this kind of refactor is a real-world performance optimization: instead of traversing our array twice, we traverse it just once. If our functions have side effects, though, they won’t execute in the same order, and order may be something we depend on! Making this kind of performance optimization in the world of side effects could break our code!
              </p><a class="display-ix08W2 position-Pjz1G2 top-kDUvL2 top-FEoZS2 visibility-PhNC81" id="hidden-outputs"></a>
              <div class="margin-left-SH03u1 margin-left-ItApa4 padding-left-5jRhu padding-left-WU6C5 position-Pjz1G2">
                <h4 class="color-BoTGL color-GOHNR margin-FBkWQ3 margin-top-EvtWd3 margin-bottom-yQEjs3 font-size-Xin5E2 font-weight-SywJI1 line-height-auF6B">Hidden outputs<a class="display-Wl0y44 display-Vk1P81 left-FBkWQ3 position-X8qzJ text-align-CdPBO2 top-RSlUg4 width-Wr637 text-decoration-gUV393 text-decoration-tfb0F1 text-decoration-Zf0gY color-Vv8Gw4 color-9rBiP color-x5AlI2 color-CwERD1 color-KmPrN color-lJpf9" href="#hidden-outputs"><svg xmlns="http://www.w3.org/2000/svg" height="20px" viewBox="0 -960 960 960" width="20px" fill="currentColor"><path d="M432-288H288q-79.68 0-135.84-56.23Q96-400.45 96-480.23 96-560 152.16-616q56.16-56 135.84-56h144v72H288q-50 0-85 35t-35 85q0 50 35 85t85 35h144v72Zm-96-156v-72h288v72H336Zm192 156v-72h144q50 0 85-35t35-85q0-50-35-85t-85-35H528v-72h144q79.68 0 135.84 56.23 56.16 56.22 56.16 136Q864-400 807.84-344 751.68-288 672-288H528Z"/></svg></a>
                </h4>
              </div>
              <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Let’s look at the simplest way to control this side effect. Rather than performing the effect in the body of the function, we can return an extra value that describes what needs to be printed. A function could print many things, so we’ll use an array of strings to model this.
              </p>
              <pre class="margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">func computeAndPrint(_ x: Int) -> (Int, [String]) {
  let computation = x * x + 1
  return (computation, ["Computed \(computation)"])
}

computeAndPrint(2)  // (5, ["Computed 5"])
</code>
              </pre>
              <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">We get a result of not only the computation, but an array of logs we may want to print.
              </p>
              <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Let’s write a test:
              </p>
              <pre class="margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1" data-line="1"><code class="language-swift highlight-pass">assertEqual(
  (5, ["Computed 5"]),
  computeAndPrint(2)
)
</code>
              </pre>
              <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Now we’re getting coverage not only on the computation, but on the effect we want to perform! Our test will now fail if the side effect is in an unexpected format.
              </p>
              <pre class="margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1" data-line="1"><code class="language-swift highlight-fail">assertEqual(
  (5, ["Computed 3"]),
  computeAndPrint(2)
)
</code>
              </pre>
              <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">The data is quite simple here, but remember, it could potentially be more critical data that describes an API request or analytics event, and we could be writing assertions that these effects are being prepared the way we expect them to.
              </p>
              <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Viewed in this way, a side effect that makes a change to the outside world is nothing but a hidden, implicit output of the function. Implicit is usually not a good thing when it comes to programming.
              </p>
              <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Now you may ask, “Well, who performs the effect?” By pulling the effect out into the return type, we have pushed responsibility for this effect to whoever calls this function. For example:
              </p>
              <pre class="margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">let (computation, logs) = computeAndPrint(2)
logs.forEach { print($0) }
</code>
              </pre>
              <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">We may not want the caller to have side effects, though, so it may have to pass them along, as well. And maybe <em>its</em> caller doesn’t want to have side effects, and so on! This sounds like a messy problem, but there are nice ways to solve it. Before we can solve <em>that</em> problem, though, we need to understand <em>this</em> one in detail.
              </p>
            </pf-vstack>
          </pf-markdown>
