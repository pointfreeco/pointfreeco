              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">We’ve now had an entire episode just about functions where we emphasized the importance of looking at the input and output types of functions to understand how they compose, but there are a lot of things a function can do that aren’t captured by its signature alone. These things are called “side effects.”
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Side effects are one of the biggest sources of complexity in code, and to make things worse they’re very difficult to test and they don’t compose well. We’ve seen from the last episode that we get a lot of benefits from embracing function composition, but side effects throw a wrench in that.
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">In this episode we’ll cover a few kinds of side effects, show why they’re so difficult to test, why they don’t compose, and try to address these problems in a nice way.
              </p>
              <div class="content-xD6dy1 margin-left-rhxaL1 display-S6VUZ flex-direction-5DQpO1 row-gap-UV0PN">
                <div>
                  <div class="font-variant-numeric-vLOwL2 line-height-B4ojl2 margin-left-xuQdp2 position-9Qeuv2 text-align-mMP5l4 width-TNVhF4 font-weight-evfWi1 line-height-9Qkso4 font-size-RTPM71" id="t43"><a data-timestamp="43" class="text-decoration-Mif7n4 text-decoration-Q5Er64 text-decoration-Zf0gY color-Vv8Gw4 color-9rBiP color-x5AlI2 color-CwERD1" href="#t43">0:43</a>
                  </div>
                </div>
              </div>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Side effect is a pretty overloaded term, so in order to define it, let’s first look at a function that has no side effects:
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">func compute(_ x: Int) -> Int {
  return x * x + 1
}
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">When we call our function, we get a result back:
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">compute(2)  // 5
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">A really nice property of functions without side effects is that no matter how many times we call one with the same input, we always get the same output:
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">compute(2)  // 5
compute(2)  // 5
compute(2)  // 5
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">This predictability makes it incredibly simple to write tests for them.
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">assertEqual(5, compute(2))  // ✅
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">If we ever write a test with the wrong expectation or the wrong input for an expectation, it will <em>always</em> fail.
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1" data-line="1-2"><code class="language-swift highlight-fail">assertEqual(4, compute(2))
assertEqual(5, compute(3))
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Let’s add a side effect to our function.
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">func computeWithEffect(_ x: Int) -> Int {
  let computation = x * x + 1
  print("Computed \(computation)")
  return computation
}
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">We’ve inserted a <code>print</code> statement right in the middle.
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">When we call <code>computeWithEffect</code> with the same input as before, we get the same output:
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">computeWithEffect(2)  // 5
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">But if we look at our console, there’s some <em>additional</em> output here.
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-txt">Computed 5
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">If we compare function signatures, <code>computeWithEffect</code> is exactly the same as <code>compute</code>, but work is being done that we couldn’t have accounted for by looking at the signature alone. The <code>print</code> function is reaching out into the world and making a change, in this case, printing to our console. Side effects require understanding the body of the function to know they’re hiding in there.
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Let’s write a test for this function:
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1" data-line="1"><code class="language-swift highlight-pass">assertEqual(5, computeWithEffect(2))
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">It passes! But now we have an <em>additional</em> line in our console.
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-txt">Computed 5
Computed 5
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">And this is behavior that we just can’t test. Here we’re just printing to the console, so it may not seem like a big deal, but if we swapped printing out for another effect, like writing to disk, making an API request, or analytics tracking, we start to care more that this behavior is happening and that we can test it.
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Side-effects can also break compositional intuitions we build. In our episode on functions, we discussed how mapping over an array with two functions is the same as mapping over an array with the composition of those functions:
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">[2, 10].map(compute).map(compute)  // [26, 10202]
[2, 10].map(compute >>> compute)   // [26, 10202]
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Now let’s try that with <code>computeWithEffect</code>:
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">[2, 10].map(computeWithEffect).map(computeWithEffect)
// [26, 10202]
[2, 10].map(computeWithEffect >>> computeWithEffect)
// [26, 10202]
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">The return values are equal, but when we look at the console, the behavior is not!
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-txt">Computed 5
Computed 101
Computed 26
Computed 10202
--
Computed 5
Computed 26
Computed 101
Computed 10202
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">We can no longer take advantage of this property without having to consider the side effects. Our ability to make this kind of refactor is a real-world performance optimization: instead of traversing our array twice, we traverse it just once. If our functions have side effects, though, they won’t execute in the same order, and order may be something we depend on! Making this kind of performance optimization in the world of side effects could break our code!
              </p><a class="content-xD6dy1 margin-left-rhxaL1 display-ix08W2 position-Pjz1G2 top-kDUvL2 top-FEoZS2 visibility-PhNC81" id="hidden-outputs"></a>
              <div class="content-xD6dy1 margin-left-rhxaL1 margin-left-SH03u1 margin-left-ItApa4 padding-left-5jRhu padding-left-WU6C5 position-Pjz1G2">
                <h4 class="color-BoTGL color-GOHNR margin-FBkWQ3 margin-top-EvtWd3 margin-bottom-yQEjs3 font-size-Xin5E2 font-weight-SywJI1 line-height-auF6B">Hidden outputs<a class="display-Wl0y44 display-Vk1P81 left-FBkWQ3 position-X8qzJ text-align-CdPBO2 top-RSlUg4 width-Wr637 text-decoration-gUV393 text-decoration-tfb0F1 text-decoration-Zf0gY color-Vv8Gw4 color-9rBiP color-x5AlI2 color-CwERD1" href="#hidden-outputs"><svg xmlns="http://www.w3.org/2000/svg" height="20px" viewBox="0 -960 960 960" width="20px" fill="currentColor"><path d="M432-288H288q-79.68 0-135.84-56.23Q96-400.45 96-480.23 96-560 152.16-616q56.16-56 135.84-56h144v72H288q-50 0-85 35t-35 85q0 50 35 85t85 35h144v72Zm-96-156v-72h288v72H336Zm192 156v-72h144q50 0 85-35t35-85q0-50-35-85t-85-35H528v-72h144q79.68 0 135.84 56.23 56.16 56.22 56.16 136Q864-400 807.84-344 751.68-288 672-288H528Z"/></svg></a>
                </h4>
              </div>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Let’s look at the simplest way to control this side effect. Rather than performing the effect in the body of the function, we can return an extra value that describes what needs to be printed. A function could print many things, so we’ll use an array of strings to model this.
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">func computeAndPrint(_ x: Int) -> (Int, [String]) {
  let computation = x * x + 1
  return (computation, ["Computed \(computation)"])
}

computeAndPrint(2)  // (5, ["Computed 5"])
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">We get a result of not only the computation, but an array of logs we may want to print.
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Let’s write a test:
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1" data-line="1"><code class="language-swift highlight-pass">assertEqual(
  (5, ["Computed 5"]),
  computeAndPrint(2)
)
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Now we’re getting coverage not only on the computation, but on the effect we want to perform! Our test will now fail if the side effect is in an unexpected format.
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1" data-line="1"><code class="language-swift highlight-fail">assertEqual(
  (5, ["Computed 3"]),
  computeAndPrint(2)
)
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">The data is quite simple here, but remember, it could potentially be more critical data that describes an API request or analytics event, and we could be writing assertions that these effects are being prepared the way we expect them to.
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Viewed in this way, a side effect that makes a change to the outside world is nothing but a hidden, implicit output of the function. Implicit is usually not a good thing when it comes to programming.
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Now you may ask, “Well, who performs the effect?” By pulling the effect out into the return type, we have pushed responsibility for this effect to whoever calls this function. For example:
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">let (computation, logs) = computeAndPrint(2)
logs.forEach { print($0) }
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">We may not want the caller to have side effects, though, so it may have to pass them along, as well. And maybe <em>its</em> caller doesn’t want to have side effects, and so on! This sounds like a messy problem, but there are nice ways to solve it. Before we can solve <em>that</em> problem, though, we need to understand <em>this</em> one in detail.
              </p>
              <div class="content-xD6dy1 margin-left-rhxaL1 display-S6VUZ flex-direction-5DQpO1 row-gap-UV0PN">
                <div>
                  <div class="font-variant-numeric-vLOwL2 line-height-B4ojl2 margin-left-xuQdp2 position-9Qeuv2 text-align-mMP5l4 width-TNVhF4 font-weight-evfWi1 line-height-9Qkso4 font-size-RTPM71" id="t495"><a data-timestamp="495" class="text-decoration-Mif7n4 text-decoration-Q5Er64 text-decoration-Zf0gY color-Vv8Gw4 color-9rBiP color-x5AlI2 color-CwERD1" href="#t495">8:15</a>
                  </div>
                </div>
              </div>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">It may seem like we’ve solved the effects problem: we just need to substitute them with descriptions in the output of our functions. Unfortunately, we’ve broken one of the most important features of functions: composition.
              </p>

              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Let’s look at a simple function that produces a greeting for a user.

              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">func uppercased(_ string: String) -> String {

              </p><a class="content-xD6dy1 margin-left-rhxaL1 display-ix08W2 position-Pjz1G2 top-kDUvL2 top-FEoZS2 visibility-PhNC81" id="mutation"></a>

              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Let’s look at a very particular type of effect and really analyze it: mutation. We’ve all had to deal with mutation in code and it can lead to a lot of complexity. Luckily, Swift provides some type-level features to help control mutation and properly document how and where it can happen.
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Here’s example of how mutation can get messy. This sample is inspired by actual code we wrote in the past, and it kept getting uglier and more of a pain to work with until we finally rewrote it to control the mutation.
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">let formatter = NumberFormatter()

func decimalStyle(_ format: NumberFormatter) {
  format.numberStyle = .decimal
  format.maximumFractionDigits = 2
}

func currencyStyle(_ format: NumberFormatter) {
  format.numberStyle = .currency
  format.roundingMode = .down
}

func wholeStyle(_ format: NumberFormatter) {
  format.maximumFractionDigits = 0
}
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Here we have a <code>NumberFormatter</code> from <code>Foundation</code> and several functions that configure number formatters with specific styles. To use these styling functions, we can apply them directly to our formatter.
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">decimalStyle(formatter)
wholeStyle(formatter)
formatter.string(for: 1234.6)  // "1,235"

currencyStyle(formatter)
formatter.string(for: 1234.6)  // "$1,234"
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">If we now re-apply our first set of formatters, we have a problem.
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">decimalStyle(formatter)
wholeStyle(formatter)
formatter.string(for: 1234.6)  // "1,234"
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">The output’s changed from <code>"1,235"</code> to <code>"1,234"</code>. The reason it’s changed? Mutation. The <code>currencyStyle</code> function’s changes have bled into other uses of our formatter, resulting in a bug that in a larger context might be hard to track down.
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">This is an example of why mutation is so tricky. It’s impossible to know what a line is doing until we trace back to learn what every line before it has done. Mutation is a manifestation of both of the side effects we’ve encountered so far, where mutable data passed between functions is a hidden input and output in one!
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">The reason we’re seeing this particular kind of mutation is because <code>NumberFormatter</code> is a “reference” type. In Swift, classes are reference types. An instance of a reference type is a single object that, when mutated, has changed for any part of the code base that holds onto or may hold onto a reference of this object. There’s no easy way to track down which parts of a code base may hold onto the same reference of an object, which can lead to a lot of confusion when mutation is involved. If our example code were used in an application and a new feature were written to depend on this formatter, subtle bugs could creep into a totally new part of the code base.
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Swift also has “value” types. This is Swift’s answer to controlling mutation. When you assign a value, you get a brand new copy to work with for the given scope. All mutations are local and anything else holding onto the same value upstream won’t see these changes.
              </p>
              <div class="content-xD6dy1 margin-left-rhxaL1 display-S6VUZ flex-direction-5DQpO1 row-gap-UV0PN">
                <div>
                  <div class="font-variant-numeric-vLOwL2 line-height-B4ojl2 margin-left-xuQdp2 position-9Qeuv2 text-align-mMP5l4 width-TNVhF4 font-weight-evfWi1 line-height-9Qkso4 font-size-RTPM71" id="t1773"><a data-timestamp="1773" class="text-decoration-Mif7n4 text-decoration-Q5Er64 text-decoration-Zf0gY color-Vv8Gw4 color-9rBiP color-x5AlI2 color-CwERD1" href="#t1773">29:33</a>
                  </div>
                </div>
              </div>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Let’s refactor this code to use values.
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">We’ll start with a struct that is a wrapper around the configuration we do with <code>NumberFormatter</code>.
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">struct NumberFormatterConfig {

  var roundingMode: NumberFormatter.RoundingMode = .up
  var maximumFractionDigits: Int = 0

  var formatter: NumberFormatter {
    let result = NumberFormatter()
    result.numberStyle = self.numberStyle
    result.roundingMode = self.roundingMode
    result.maximumFractionDigits = self.maximumFractionDigits
    return result
  }
}
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">It has some nice defaults and a <code>formatter</code> computed property that we can use to derive new, “honest” <code>NumberFormatter</code>s. What does it look like to update our styling functions to use <code>NumberFormatterConfig</code> instead?
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">func decimalStyle(
  _ format: NumberFormatterConfig
) -> NumberFormatterConfig {
  var format = format
  format.numberStyle = .decimal
  format.maximumFractionDigits = 2
  return format
}

func currencyStyle(
  _ format: NumberFormatterConfig
) -> NumberFormatterConfig {
  var format = format
  format.numberStyle = .currency
  format.roundingMode = .down
  return format
}

func wholeStyle(
  _ format: NumberFormatterConfig
) -> NumberFormatterConfig {
  var format = format
  format.maximumFractionDigits = 0
  return format
}
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Each styling function takes a <code>NumberFormatterConfig</code>, copies it using the <code>var</code> keyword, and mutates the local copy before returning it to the caller.
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Using it looks a bit different.
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">let config = NumberFormatterConfig()

wholeStyle(decimalStyle(config))
  .formatter
  .string(for: 1234.6)
// "1,235"

currencyStyle(config)
  .formatter
  .string(for: 1234.6)
// "$1,234"

wholeStyle(decimalStyle(config))
  .formatter
  .string(for: 1234.6)
// "1,235"
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">This time, whenever we pass <code>config</code> to a styling function, we get a brand new copy, and our bug goes away!
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">We could have done something similar with reference types using the <code>copy</code> method on classes that implement <code>NSCopying</code> and return this copy explicitly:
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">func decimalStyle(_ format: NumberFormatter) -> NumberFormatter {
  let format = format.copy() as! NumberFormatter
  format.numberStyle = .decimal
  format.maximumFractionDigits = 2
  return format
}
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Unfortunately here, the compiler gives us no guarantees that we don’t mutate the original formatter. On top of that, the caller may expect a copy, feel free to make further mutations, and the complexity builds from there!
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Because reference types aren’t automatically copied, they do have some nice performance benefits. Luckily, Swift provides a nice, semantic way of mutating values in place: the <code>inout</code> keyword.
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Let’s modify our config styling function to use <code>inout</code>.
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">func inoutDecimalStyle(_ format: inout NumberFormatterConfig) {
  format.numberStyle = .decimal
  format.maximumFractionDigits = 2
}

func inoutCurrencyStyle(_ format: inout NumberFormatterConfig) {
  format.numberStyle = .currency
  format.roundingMode = .down
}

func inoutWholeStyle(_ format: inout NumberFormatterConfig) {
  format.maximumFractionDigits = 0
}
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">This looks a lot like our original, mutating functions on <code>NumberFormatter</code>. We can perform mutations right away and don’t have to worry about copying values or returning them. Let’s try using these styling functions the same way we used the original ones.
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">let config = NumberFormatterConfig()

inoutDecimalStyle(config)
inoutWholeStyle(config)
config.formatter.string(from: 1234.6)
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">We get a compiler error!
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-txt">Cannot pass immutable value as inout argument: 'config' is a 'let' constant
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Swift even offers to fix this for us, transforming <code>let</code> into <code>var</code>.
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">var config = NumberFormatterConfig()

inoutDecimalStyle(config)
inoutWholeStyle(config)
config.formatter.string(from: 1234.6)
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">But that wasn’t enough. We have another compiler error!
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-txt">Passing value of type 'NumberFormatterConfig' to an inout parameter requires explicit '&amp;'
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Swift requires us to annotate, at the call site, that we’re agreeing to let this data be mutated.
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">inoutDecimalStyle(&amp;config)
inoutWholeStyle(&amp;config)
config.formatter.string(from: 1234.6)  // "1,235"
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">We can continue to call our mutating style functions in a similar fashion.
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">inoutCurrencyStyle(&amp;config)
config.formatter.string(from: 1234.6)  // "$1,234"

inoutDecimalStyle(&amp;config)
inoutWholeStyle(&amp;config)
config.formatter.string(from: 1234.6)  // "1,234"
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">And there’s our bug again, but the code involved now has a lot of syntax that screams “mutation” and it makes this kind of bug much easier to track down.
              </p>
              <div class="content-xD6dy1 margin-left-rhxaL1 display-S6VUZ flex-direction-5DQpO1 row-gap-UV0PN">
                <div>
                  <div class="font-variant-numeric-vLOwL2 line-height-B4ojl2 margin-left-xuQdp2 position-9Qeuv2 text-align-mMP5l4 width-TNVhF4 font-weight-evfWi1 line-height-9Qkso4 font-size-RTPM71" id="t2082"><a data-timestamp="2082" class="text-decoration-Mif7n4 text-decoration-Q5Er64 text-decoration-Zf0gY color-Vv8Gw4 color-9rBiP color-x5AlI2 color-CwERD1" href="#t2082">34:42</a>
                  </div>
                </div>
              </div>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">It’s great that Swift provides a nice solution to mutation problems by providing type-level features to control where mutation can happen and how far it can travel. We still have a problem to solve if we want to use it, though.
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">The styling functions we used that returned brand new copies have a nice shape:
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">(NumberFormatterConfig) -> NumberFormatterConfig
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">They have the same input and output which means they compose with each other, <em>and</em> they compose with any other functions that return or take a <code>NumberFormatterConfig</code>!
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">decimalStyle >>> currencyStyle
// (NumberFormatterConfig) -> NumberFormatterConfig
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Now we have a brand new styling function that we’ve composed from smaller pieces.
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Meanwhile, our <code>inout</code> functions don’t have this shape: their inputs and outputs don’t match and they don’t compose with many functions in general. These functions have the same logic, though, so there must be a way to bridge the <code>inout</code> world with the function world.
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">It turns out that we can define a function called <code>toInout</code> that converts a function with the same input and output type into an <code>inout</code> function.
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">func toInout&lt;A>(
  _ f: @escaping (A) -> A
) -> ((inout A) -> Void) {

  return { a in
    a = f(a)
  }
}
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">We can also define a dual function, <code>fromInout</code>, that does the inverse transformation.
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">func fromInout&lt;A>(
  _ f: @escaping (inout A) -> Void
) -> ((A) -> A) {

  return { a in
    var copy = a
    f(&amp;copy)
    return copy
  }
}
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">What we see here is there’s a natural correspondence between <code>(A) -> A</code> functions and <code>(inout A) -> Void</code> functions. Functions from <code>(A) -> A</code> compose very nicely, so through this correspondence, we would hope that functions from <code>(inout A) -> Void</code> could share these compositional qualities.
              </p><a class="content-xD6dy1 margin-left-rhxaL1 display-ix08W2 position-Pjz1G2 top-kDUvL2 top-FEoZS2 visibility-PhNC81" id="introducing"></a>
              <div class="content-xD6dy1 margin-left-rhxaL1 margin-left-SH03u1 margin-left-ItApa4 padding-left-5jRhu padding-left-WU6C5 position-Pjz1G2">
                <h4 class="color-BoTGL color-GOHNR margin-FBkWQ3 margin-top-EvtWd3 margin-bottom-yQEjs3 font-size-Xin5E2 font-weight-SywJI1 line-height-auF6B">Introducing &lt;><a class="display-Wl0y44 display-Vk1P81 left-FBkWQ3 position-X8qzJ text-align-CdPBO2 top-RSlUg4 width-Wr637 text-decoration-gUV393 text-decoration-tfb0F1 text-decoration-Zf0gY color-Vv8Gw4 color-9rBiP color-x5AlI2 color-CwERD1" href="#introducing"><svg xmlns="http://www.w3.org/2000/svg" height="20px" viewBox="0 -960 960 960" width="20px" fill="currentColor"><path d="M432-288H288q-79.68 0-135.84-56.23Q96-400.45 96-480.23 96-560 152.16-616q56.16-56 135.84-56h144v72H288q-50 0-85 35t-35 85q0 50 35 85t85 35h144v72Zm-96-156v-72h288v72H336Zm192 156v-72h144q50 0 85-35t35-85q0-50-35-85t-85-35H528v-72h144q79.68 0 135.84 56.23 56.16 56.22 56.16 136Q864-400 807.84-344 751.68-288 672-288H528Z"/></svg></a>
                </h4>
              </div>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Even though we saw that <code>(A) -> A</code> functions compose using <code>>>></code>, we shouldn’t reuse this operator because it has way too much freedom. We’re looking at a much more constrained, single-type composition. So let’s define a new operator. Let’s start with the precedence group.
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">precedencegroup SingleTypeComposition {
  associativity: left
  higherThan: ForwardApplication
}
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">And let’s define our operator.
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">infix operator &lt;>: SingleTypeComposition
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">A fun name for this operator is the “diamond” operator.
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">We can define the operator against <code>(A) -> A</code> simply enough:
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">func &lt;> &lt;A>(
  f: @escaping (A) -> A,
  g: @escaping (A) -> A
) -> ((A) -> A) {

  return f >>> g
}
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">It may looks silly to have just wrapped one operator with another, but we’ve constrained how it can be used and encoded some meaning: when we see this operator we know we’re dealing with a single type!
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Let’s define <code>&lt;></code> for <code>inout</code> functions:
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">func &lt;> &lt;A>(
  f: @escaping (inout A) -> Void,
  g: @escaping (inout A) -> Void
) -> ((inout A) -> Void) {

  return { a in
    f(&amp;a)
    g(&amp;a)
  }
}
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Our previous composition works.
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">decimalStyle &lt;> currencyStyle
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Better yet, we can compose our <code>inout</code> styling functions!
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">inoutDecimalStyle &lt;> inoutCurrencyStyle
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">What happens when we start piping values into our compositions?
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">config |> decimalStyle &lt;> currencyStyle
config |> inoutDecimalStyle &lt;> inoutCurrencyStyle
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Our <code>inout</code> version produces an error.
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-txt">Cannot convert value of type '(inout Int) -> ()' to expected argument type '(_) -> _'
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">This is because <code>|></code> doesn’t yet work in the world of <code>inout</code>, but we can define an overload that does.
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">func |> &lt;A>(a: inout A, f: (inout A) -> Void) -> Void {
  f(&amp;a)
}
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">And now we can freely pipe values into these mutable pipelines.
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">config |> inoutDecimalStyle &lt;> inoutCurrencyStyle
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">This is great! We don’t have to sacrifice composability to take advantage of some nice Swift features.
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">We’ve solved this problem at the cost of yet another operator, so it’s time to check the boxes.
              </p>
              <ol class="content-xD6dy1 margin-left-rhxaL1 display-BvS8W3 flex-direction-7gclL row-gap-NKv2f3">
                <li>
                  <pf-vstack class="align-items-msN8p3 display-BvS8W3 flex-direction-7gclL max-width-C8uWv row-gap-NKv2f3">
                    <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Does this operator exist in Swift? Nope, so no potential for confusion there.
                    </p>
                  </pf-vstack>
                </li>
                <li>
                  <pf-vstack class="align-items-msN8p3 display-BvS8W3 flex-direction-7gclL max-width-C8uWv row-gap-NKv2f3">
                    <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Is there prior art? Yep. It exists in Haskell, PureScript, and other languages with strong functional communities that have adopted it. It has a nice shape that points in both directions and kind of signals a joining together.
                    </p>
                  </pf-vstack>
                </li>
                <li>
                  <pf-vstack class="align-items-msN8p3 display-BvS8W3 flex-direction-7gclL max-width-C8uWv row-gap-NKv2f3">
                    <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Is this a universal operator or is it only solving a domain-specific problem? We’ve only defined this operator for <code>(A) -> A</code> functions and <code>(inout A) -> Void</code> functions so far, but it turns out that <code>&lt;></code> is used far more generally for combining two things of the same type into one, which is kind of the most fundamental unit of computation there is. We’re going to encounter this operator all over the place.
                    </p>
                  </pf-vstack>
                </li>
              </ol><a class="content-xD6dy1 margin-left-rhxaL1 display-ix08W2 position-Pjz1G2 top-kDUvL2 top-FEoZS2 visibility-PhNC81" id="what-s-the-point"></a>

                <h4 class="color-BoTGL color-GOHNR margin-FBkWQ3 margin-top-EvtWd3 margin-bottom-yQEjs3 font-size-Xin5E2 font-weight-SywJI1 line-height-auF6B">What’s the point?<a class="display-Wl0y44 display-Vk1P81 left-FBkWQ3 position-X8qzJ text-align-CdPBO2 top-RSlUg4 width-Wr637 text-decoration-gUV393 text-decoration-tfb0F1 text-decoration-Zf0gY color-Vv8Gw4 color-9rBiP color-x5AlI2 color-CwERD1" href="#what-s-the-point"><svg xmlns="http://www.w3.org/2000/svg" height="20px" viewBox="0 -960 960 960" width="20px" fill="currentColor"><path d="M432-288H288q-79.68 0-135.84-56.23Q96-400.45 96-480.23 96-560 152.16-616q56.16-56 135.84-56h144v72H288q-50 0-85 35t-35 85q0 50 35 85t85 35h144v72Zm-96-156v-72h288v72H336Zm192 156v-72h144q50 0 85-35t35-85q0-50-35-85t-85-35H528v-72h144q79.68 0 135.84 56.23 56.16 56.22 56.16 136Q864-400 807.84-344 751.68-288 672-288H528Z"/></svg></a>
                </h4>
              </div>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">It’s time to slow down and ask: “What’s the point?” We’ve encountered a lot of effects that introduced complexity into our code and made it more difficult to test. We decided to fix this by doing a little bit of upfront work to make the effects explicit in the types, both inputs and outputs, but then we broke composition. Then we introduced operators that aid in composition specifically for composing effects. Was it worth it?
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">We’d say so! We were able to lift our effectful code, which was untestable and difficult to reason about in isolation, up into a world where effects were explicit and we could test and understand a single line without understanding any of the lines that came before it. We did all of this without breaking composition. That’s really powerful!
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Meanwhile, our extra bit of upfront work should save us an immeasurable amount of time spent debugging code that has a complex web of mutation, time fixing bugs in side effects that are sprinkled everywhere, and time jumping through hoops to make code testable.
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Side effects are a <em>huge</em> topic, and we’ve only scratched the surface. We’re going to be exploring lots of interesting ways to control side effects in the future. Stay tuned!
              </p>
