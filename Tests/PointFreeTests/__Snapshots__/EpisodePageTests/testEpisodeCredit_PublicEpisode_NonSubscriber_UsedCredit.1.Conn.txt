              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">When our operator is used multiple times in a row, Swift doesn’t know which side of the operator to evaluate first. On the lefthand side, we have:
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">2 |> incr
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3"><code>2</code> piped into <code>incr</code> makes sense, since <code>incr</code> takes an integer. On the righthand side, we have:
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">incr |> square
</code>
              </pre>
              <div class="content-xD6dy1 margin-left-rhxaL1 display-S6VUZ flex-direction-5DQpO1 row-gap-UV0PN">
                <div>
                  <div class="font-variant-numeric-vLOwL2 line-height-B4ojl2 margin-left-xuQdp2 position-9Qeuv2 text-align-mMP5l4 width-TNVhF4 font-weight-evfWi1 line-height-9Qkso4 font-size-RTPM71" id="t230"><a data-timestamp="230" class="text-decoration-Mif7n4 text-decoration-Q5Er64 text-decoration-Zf0gY color-Vv8Gw4 color-9rBiP color-x5AlI2 color-CwERD1" href="#t230">3:50</a>
                  </div>
                </div>
              </div>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Piping our <code>incr</code> function into our <code>square</code> function makes less sense: <code>square</code> expects an integer, not a function.
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">We need to give Swift a hint as to which expression to evaluate first. One way to do this is to wrap the lefthand expression with parentheses so it evaluates first.
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">(2 |> incr) |> square  // 9
</code>
              </pre>
              <div class="content-xD6dy1 margin-left-rhxaL1 display-S6VUZ flex-direction-5DQpO1 row-gap-UV0PN">
                <div>
                  <div class="font-variant-numeric-vLOwL2 line-height-B4ojl2 margin-left-xuQdp2 position-9Qeuv2 text-align-mMP5l4 width-TNVhF4 font-weight-evfWi1 line-height-9Qkso4 font-size-RTPM71" id="t240"><a data-timestamp="240" class="text-decoration-Mif7n4 text-decoration-Q5Er64 text-decoration-Zf0gY color-Vv8Gw4 color-9rBiP color-x5AlI2 color-CwERD1" href="#t240">4:00</a>
                  </div>
                </div>
              </div>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">This works but it’s kind of messy. This is a pretty simple composition, but a more complicated example would require more nested parentheses and would be more difficult to follow. Let’s give Swift a better hint.
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Swift lets us define the associativity of an operator by using a precedence group. Let’s define a precedence group for function application:
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">precedencegroup ForwardApplication {
  associativity: left
}
</code>
              </pre>
              <div class="content-xD6dy1 margin-left-rhxaL1 display-S6VUZ flex-direction-5DQpO1 row-gap-UV0PN">
                <div>
                  <div class="font-variant-numeric-vLOwL2 line-height-B4ojl2 margin-left-xuQdp2 position-9Qeuv2 text-align-mMP5l4 width-TNVhF4 font-weight-evfWi1 line-height-9Qkso4 font-size-RTPM71" id="t260"><a data-timestamp="260" class="text-decoration-Mif7n4 text-decoration-Q5Er64 text-decoration-Zf0gY color-Vv8Gw4 color-9rBiP color-x5AlI2 color-CwERD1" href="#t260">4:20</a>
                  </div>
                </div>
              </div>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">We gave it a left associativity to ensure that the lefthand expression evaluates first.
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Now we need to make sure our operator conforms to our precedence group.
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">infix operator |>: ForwardApplication
</code>
              </pre>
              <div class="content-xD6dy1 margin-left-rhxaL1 display-S6VUZ flex-direction-5DQpO1 row-gap-UV0PN">
                <div>
                  <div class="font-variant-numeric-vLOwL2 line-height-B4ojl2 margin-left-xuQdp2 position-9Qeuv2 text-align-mMP5l4 width-TNVhF4 font-weight-evfWi1 line-height-9Qkso4 font-size-RTPM71" id="t275"><a data-timestamp="275" class="text-decoration-Mif7n4 text-decoration-Q5Er64 text-decoration-Zf0gY color-Vv8Gw4 color-9rBiP color-x5AlI2 color-CwERD1" href="#t275">4:35</a>
                  </div>
                </div>
              </div>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">We can now get rid of our parentheses.
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">2 |> incr |> square  // 9
</code>
              </pre>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">This looks a lot like our earlier method version:
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">2.incr().square()  // 9
</code>

              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">We’ve solved the readability problem of nested functions, but we have a new problem: custom operators. Custom operators aren’t very common. In fact, many of us avoid them because they have a bad reputation. A typical encounter with custom operators is within a subset of the idea: overloaded operators.
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">For example, in C++ you can’t define new operators, but you can overload existing operators the language provides. If we were to write a vector library in C++, we could overload <code>+</code> to mean the sum of two vectors and we could overload <code>*</code> to mean the dot product of two vectors. Then again, <code>*</code> could mean the cross product of two vectors, so any choice to use <code>*</code> is going to require an opinion and lead to confusion.
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">We would never suggest overloading multiplication with function application:
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">2 * incr  // What does this mean!?
</code>
              </pre>
              <div class="content-xD6dy1 margin-left-rhxaL1 display-S6VUZ flex-direction-5DQpO1 row-gap-UV0PN">
                <div>

                  </div>
                </div>
              </div>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">It would be very difficult to encounter this in a code base and understand what it means.
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Luckily we don’t have this problem here. We’re using a brand new operator, <code>|></code>, which Swift has no prior knowledge about. One may argue that an operator that Swift doesn’t know about is an operator that a Swift developer doesn’t know about, but in this case, we’re looking to prior art: F#, Elixir, and Elm all use this operator in the same way. Swift engineers that are familiar with these languages are going to be familiar with the operator. It also has a nice shape! The pipe (<code>|</code>) evokes Unix, where we pipe the output of programs as input into other programs. The arrow also goes to the right (<code>></code>), which gives us a nice left-to-right reading. Let’s look back at our usage of this operator:
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">2 |> incr
</code>

                </div>
              </div>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Even if we’re not familiar with this operator, we can kind of piece together what’s going on here.
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">We’re going to be using operators a lot in Point-Free, so let’s make sure that we’re responsible and justify the symbols that we introduce. There are a few boxes we’ll want tick before we should introduce a new operator:
              </p>
              <ol class="content-xD6dy1 margin-left-rhxaL1 display-BvS8W3 flex-direction-7gclL row-gap-NKv2f3">
                <li>
                  <pf-vstack class="align-items-msN8p3 display-BvS8W3 flex-direction-7gclL max-width-C8uWv row-gap-NKv2f3">
                    <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">We shouldn’t overload an operator with existing meaning with new meaning.
                    </p>
                  </pf-vstack>
                </li>
                <li>
                  <pf-vstack class="align-items-msN8p3 display-BvS8W3 flex-direction-7gclL max-width-C8uWv row-gap-NKv2f3">
                    <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">We should leverage prior art as much as possible and make sure our operator has a nice “shape” that evokes its semantics: in this case <code>|></code> nicely describes piping a value forward into a function.
                    </p>
                  </pf-vstack>
                </li>
                <li>
                  <pf-vstack class="align-items-msN8p3 display-BvS8W3 flex-direction-7gclL max-width-C8uWv row-gap-NKv2f3">
                    <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">We shouldn’t be inventing operators to solve very domain-specific problems. We should only introduce operators that can be used and reused in very general ways.
                    </p>
                  </pf-vstack>
                </li>
              </ol>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Our <code>|></code> operator ticks all these boxes.

              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Meanwhile, there’s something in the free function world that isn’t possible in the method world: function composition. Function composition is the ability to take two functions where the output of one matches the input of another so that we can glue them together and get a whole new function. In order to embrace this, we’re going to introduce another operator:
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">infix operator >>>
</code>
              </pre>
              <div class="content-xD6dy1 margin-left-rhxaL1 display-S6VUZ flex-direction-5DQpO1 row-gap-UV0PN">

                  <div class="font-variant-numeric-vLOwL2 line-height-B4ojl2 margin-left-xuQdp2 position-9Qeuv2 text-align-mMP5l4 width-TNVhF4 font-weight-evfWi1 line-height-9Qkso4 font-size-RTPM71" id="t565"><a data-timestamp="565" class="text-decoration-Mif7n4 text-decoration-Q5Er64 text-decoration-Zf0gY color-Vv8Gw4 color-9rBiP color-x5AlI2 color-CwERD1" href="#t565">9:25</a>
                  </div>
                </div>
              </div>

              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Because of this, they’re less reusable by default!
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">While it can feel like we generally work with methods, not functions, in Swift, we use functions every day and we may not even think about it.
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">One very common function that we use every day is the initializer! It’s a global function that produces a value. And all of Swift’s initializers are at our disposal for function composition. We can take a previous composition and forward-compose it into a <code>String</code> initializer.
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">incr >>> square >>> String.init
// (Int) -> String
</code>
              </pre>
              <div class="content-xD6dy1 margin-left-rhxaL1 display-S6VUZ flex-direction-5DQpO1 row-gap-UV0PN">
                <div>
                  <div class="font-variant-numeric-vLOwL2 line-height-B4ojl2 margin-left-xuQdp2 position-9Qeuv2 text-align-mMP5l4 width-TNVhF4 font-weight-evfWi1 line-height-9Qkso4 font-size-RTPM71" id="t924"><a data-timestamp="924" class="text-decoration-Mif7n4 text-decoration-Q5Er64 text-decoration-Zf0gY color-Vv8Gw4 color-9rBiP color-x5AlI2 color-CwERD1" href="#t924">15:24</a>
                  </div>
                </div>
              </div>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">And we can pipe a value through to produce a string result.
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">2 |> incr >>> square >>> String.init  // "9"
</code>
              </pre>
              <div class="content-xD6dy1 margin-left-rhxaL1 display-S6VUZ flex-direction-5DQpO1 row-gap-UV0PN">
                <div>
                  <div class="font-variant-numeric-vLOwL2 line-height-B4ojl2 margin-left-xuQdp2 position-9Qeuv2 text-align-mMP5l4 width-TNVhF4 font-weight-evfWi1 line-height-9Qkso4 font-size-RTPM71" id="t943"><a data-timestamp="943" class="text-decoration-Mif7n4 text-decoration-Q5Er64 text-decoration-Zf0gY color-Vv8Gw4 color-9rBiP color-x5AlI2 color-CwERD1" href="#t943">15:43</a>
                  </div>
                </div>
              </div>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Meanwhile, in the method world, we can’t chain the result along to our initializer. We need to change the order in which we read things by wrapping the initializer around the methods.
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">String(2.incr().square())
</code>
              </pre>
              <div class="content-xD6dy1 margin-left-rhxaL1 display-S6VUZ flex-direction-5DQpO1 row-gap-UV0PN">
                <div>
                  <div class="font-variant-numeric-vLOwL2 line-height-B4ojl2 margin-left-xuQdp2 position-9Qeuv2 text-align-mMP5l4 width-TNVhF4 font-weight-evfWi1 line-height-9Qkso4 font-size-RTPM71" id="t951"><a data-timestamp="951" class="text-decoration-Mif7n4 text-decoration-Q5Er64 text-decoration-Zf0gY color-Vv8Gw4 color-9rBiP color-x5AlI2 color-CwERD1" href="#t951">15:51</a>
                  </div>
                </div>
              </div>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">In addition to there being a lot of free functions that initializers give us, there are also a ton of functions in the standard library that take free functions as input. On <code>Array</code>, we have a method called <code>map</code>:
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">[1, 2, 3].map
// (transform: (Int) throws -> T) rethrows -> [T]
</code>
              </pre>
              <div class="content-xD6dy1 margin-left-rhxaL1 display-S6VUZ flex-direction-5DQpO1 row-gap-UV0PN">
                <div>
                  <div class="font-variant-numeric-vLOwL2 line-height-B4ojl2 margin-left-xuQdp2 position-9Qeuv2 text-align-mMP5l4 width-TNVhF4 font-weight-evfWi1 line-height-9Qkso4 font-size-RTPM71" id="t961"><a data-timestamp="961" class="text-decoration-Mif7n4 text-decoration-Q5Er64 text-decoration-Zf0gY color-Vv8Gw4 color-9rBiP color-x5AlI2 color-CwERD1" href="#t961">16:01</a>
                  </div>
                </div>
              </div>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">This method takes a free function from the array’s element type to another type <code>T</code> and transforms every element to return a new array of <code>T</code>s.
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Typically, we pass an ad hoc function here. For example, we could increment and square:
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">[1, 2, 3].map { ($0 + 1) * ($0 + 1) }  // [4, 9, 16]
</code>
              </pre>
              <div class="content-xD6dy1 margin-left-rhxaL1 display-S6VUZ flex-direction-5DQpO1 row-gap-UV0PN">
                <div>
                  <div class="font-variant-numeric-vLOwL2 line-height-B4ojl2 margin-left-xuQdp2 position-9Qeuv2 text-align-mMP5l4 width-TNVhF4 font-weight-evfWi1 line-height-9Qkso4 font-size-RTPM71" id="t986"><a data-timestamp="986" class="text-decoration-Mif7n4 text-decoration-Q5Er64 text-decoration-Zf0gY color-Vv8Gw4 color-9rBiP color-x5AlI2 color-CwERD1" href="#t986">16:26</a>
                  </div>
                </div>
              </div>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">When we only work with methods, we seem to avoid reusability. We’re working with functions, though, and we can reuse them directly.
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">[1, 2, 3]
  .map(incr)
  .map(square)
// [4, 9, 16]
</code>
              </pre>
              <div class="content-xD6dy1 margin-left-rhxaL1 display-S6VUZ flex-direction-5DQpO1 row-gap-UV0PN">
                <div>
                  <div class="font-variant-numeric-vLOwL2 line-height-B4ojl2 margin-left-xuQdp2 position-9Qeuv2 text-align-mMP5l4 width-TNVhF4 font-weight-evfWi1 line-height-9Qkso4 font-size-RTPM71" id="t999"><a data-timestamp="999" class="text-decoration-Mif7n4 text-decoration-Q5Er64 text-decoration-Zf0gY color-Vv8Gw4 color-9rBiP color-x5AlI2 color-CwERD1" href="#t999">16:39</a>
                  </div>
                </div>
              </div>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">We didn’t have to open up a new ad hoc function or specify arguments, and this is known as “point-free” style. When we define functions and specify arguments, even <code>$0</code>, these arguments are known as “points”. Programming in the “point-free” style emphasizes a focus on functions and composition so that we don’t have to even refer to the data being operated on. That’s what this series was named after!
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Mapping over our array with <code>square</code> before mapping over the resulting array with <code>incr</code> is equivalent to function composition! We can just map once and forward-compose <code>incr</code> into <code>square</code>:
              </p>
              <pre class="content-xD6dy1 margin-left-rhxaL1 margin-FBkWQ3 margin-bottom-NKv2f3 overflow-x-u7yQf2 padding-wKavA3 border-radius-xypJ01 color-K2J0r1 color-jj5LZ background-color-c5ixr2 background-color-xnERS1"><code class="language-swift">[1, 2, 3].map(incr >>> square)  // [4, 9, 16]
</code>
              </pre>
              <div class="content-xD6dy1 margin-left-rhxaL1 display-S6VUZ flex-direction-5DQpO1 row-gap-UV0PN">
                <div>
                  <div class="font-variant-numeric-vLOwL2 line-height-B4ojl2 margin-left-xuQdp2 position-9Qeuv2 text-align-mMP5l4 width-TNVhF4 font-weight-evfWi1 line-height-9Qkso4 font-size-RTPM71" id="t1051"><a data-timestamp="1051" class="text-decoration-Mif7n4 text-decoration-Q5Er64 text-decoration-Zf0gY color-Vv8Gw4 color-9rBiP color-x5AlI2 color-CwERD1" href="#t1051">17:31</a>
                  </div>
                </div>
              </div>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">This is really cool! We can see a relationship that our composition operator has with <code>map</code> that would have been difficult to see with methods. Here, <code>map</code> distributes over <code>>>></code> composition: the composition of <code>map</code> twice over two functions is the composition of two functions over a <code>map</code>. There are a lot of patterns like this and we’ll be exploring them in the future!
              </p><a class="content-xD6dy1 margin-left-rhxaL1 display-ix08W2 position-Pjz1G2 top-kDUvL2 top-FEoZS2 visibility-PhNC81" id="what-s-the-point"></a>
              <div class="content-xD6dy1 margin-left-rhxaL1 margin-left-SH03u1 margin-left-ItApa4 padding-left-5jRhu padding-left-WU6C5 position-Pjz1G2">
                <h4 class="color-BoTGL color-GOHNR margin-FBkWQ3 margin-top-EvtWd3 margin-bottom-yQEjs3 font-size-Xin5E2 font-weight-SywJI1 line-height-auF6B">What’s the point?<a class="display-Wl0y44 display-Vk1P81 left-FBkWQ3 position-X8qzJ text-align-CdPBO2 top-RSlUg4 width-Wr637 text-decoration-gUV393 text-decoration-tfb0F1 text-decoration-Zf0gY color-Vv8Gw4 color-9rBiP color-x5AlI2 color-CwERD1" href="#what-s-the-point"><svg xmlns="http://www.w3.org/2000/svg" height="20px" viewBox="0 -960 960 960" width="20px" fill="currentColor"><path d="M432-288H288q-79.68 0-135.84-56.23Q96-400.45 96-480.23 96-560 152.16-616q56.16-56 135.84-56h144v72H288q-50 0-85 35t-35 85q0 50 35 85t85 35h144v72Zm-96-156v-72h288v72H336Zm192 156v-72h144q50 0 85-35t35-85q0-50-35-85t-85-35H528v-72h144q79.68 0 135.84 56.23 56.16 56.22 56.16 136Q864-400 807.84-344 751.68-288 672-288H528Z"/></svg></a>
                </h4>
              </div>
              <div class="content-xD6dy1 margin-left-rhxaL1 display-S6VUZ flex-direction-5DQpO1 row-gap-UV0PN">
                <div>
                  <div class="font-variant-numeric-vLOwL2 line-height-B4ojl2 margin-left-xuQdp2 position-9Qeuv2 text-align-mMP5l4 width-TNVhF4 font-weight-evfWi1 line-height-9Qkso4 font-size-RTPM71" id="t1113"><a data-timestamp="1113" class="text-decoration-Mif7n4 text-decoration-Q5Er64 text-decoration-Zf0gY color-Vv8Gw4 color-9rBiP color-x5AlI2 color-CwERD1" href="#t1113">18:33</a>
                  </div>
                </div>
              </div>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Let’s slow down and ask ourselves: <em>what’s the point?</em> Why are we doing all of this? In this episode we’ve introduced <em>two</em> custom operators and polluted the global namespace with free functions. Why not continue to use the methods we know and love?
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Hopefully the code we’ve written today has made a strong case for introducing functions into our workflows: functions compose in ways that methods cannot. Composing functionality with methods requires a lot more work and boilerplate, and trying to see that composition afterwards requires filtering that noise. With just a couple operators, we unlock a world of composition that we didn’t have before, and we retain a lot of the readability we expect!
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">Swift also doesn’t really have a “global namespace” that we need to be worried about. We can scope our functions in a lot of different ways:
              </p>
              <ul class="content-xD6dy1 margin-left-rhxaL1 margin-bottom-FBkWQ3 margin-top-FBkWQ3 display-BvS8W3 flex-direction-7gclL row-gap-NKv2f3">
                <li>
                  <pf-vstack class="align-items-msN8p3 display-BvS8W3 flex-direction-7gclL max-width-C8uWv row-gap-NKv2f3">
                    <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">We can define functions that are private to a file.
                    </p>
                  </pf-vstack>
                </li>
                <li>
                  <pf-vstack class="align-items-msN8p3 display-BvS8W3 flex-direction-7gclL max-width-C8uWv row-gap-NKv2f3">
                    <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">We can define functions that are static members on structs and enums.
                    </p>
                  </pf-vstack>
                </li>
                <li>
                  <pf-vstack class="align-items-msN8p3 display-BvS8W3 flex-direction-7gclL max-width-C8uWv row-gap-NKv2f3">
                    <p class="line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">We can define functions that are scoped to modules. We can use several libraries that define the same function name, but qualify them by the library’s module name.
                    </p>
                  </pf-vstack>
                </li>
              </ul>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">I think it’s safe to say: “Don’t fear the function.”
              </p>
              <p class="content-xD6dy1 margin-left-rhxaL1 line-height-9Qkso4 padding-FBkWQ3 margin-FBkWQ3">We’re going to be using functions a lot on Point-Free. It’s hard to imagine an episode in which we’re not going to be using free functions. We’ll be building very complex systems that under the hood are just functions and composition. It’s really beautiful and exciting to see how it all works and how everything pieces together. Function composition will continue to help us see things that we couldn’t have seen without it.
              </p>
